The FamilyCo server exposes a Fastify HTTP API. Every workflow surface ‚Äî chat, agents, projects, tasks, inbox, audit, budget, skills, tools, plugins, automation, providers ‚Äî has a corresponding controller under `apps/server/src/modules/<module>`.

## Base URL

| Mode | Base URL |
| --- | --- |
| Local server | `http://127.0.0.1:4000` |
| Electron desktop | The embedded server picks a free port at startup. |

All routes live under the `API_PREFIX` (default `/api/v1`).

## Authentication

The API supports two auth modes. Both are described in detail in [Auth and access control](/modules/auth).

### API key

Send your key in the `Authorization` header:

```http
Authorization: Bearer <FAMILYCO_API_KEY>
```

API keys are stored as `ApiKeyRecord` rows. Only the hash is persisted; the raw value is returned exactly once at creation. The `API_KEY_SALT` env var is used for hashing ‚Äî see [Configuration](/guides/configuration).

### JWT

Mint a short-lived JWT from an API key, then use it like a bearer token:

```http
Authorization: Bearer <jwt>
```

JWTs are signed with `JWT_SECRET`.

## Authorization

- All non-token routes require API key or JWT authentication.
- Route access enforces a minimum agent **level**: `L0` (executive), `L1` (manager), or `L2` (worker).
- API key lifecycle operations (create / revoke / rotate) are auditable.

## Read-only mode

If startup migration safety fails, the server enters read-only mode. Mutation requests return:

```json
{
  "statusCode": 503,
  "code": "READ_ONLY_MODE",
  "message": "Application is in read-only mode because database migration failed"
}
```

The health endpoint reports queue stats, migration status, and read-only mode.

## Module layout

The server registers controllers per module. The current set:

- `auth` ‚Äî API keys, JWT issuance.
- `agent` ‚Äî agent management and chat.
- `approval` ‚Äî approvals and resolutions.
- `audit` ‚Äî audit log queries.
- `budget` ‚Äî usage and caps.
- `cron` ‚Äî recurring automation.
- `dashboard` ‚Äî aggregated views.
- `engine` ‚Äî queue and run inspection.
- `inbox` ‚Äî Founder queue.
- `knowledge` ‚Äî knowledge artifacts.
- `plugins` ‚Äî plugin discovery and enablement.
- `project` ‚Äî project CRUD.
- `provider` ‚Äî provider connections.
- `settings` ‚Äî settings store.
- `setup` ‚Äî first-run setup.
- `skills` ‚Äî skill discovery and toggling.
- `task` ‚Äî task CRUD and execution.
- `tools` ‚Äî tool policy and custom fields.

For per-module behavior, contracts, and constraints, see the [Modules](/modules/chat) section.
